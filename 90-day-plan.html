90-day entry plan
Entry principle: Earn the right to standardize by shipping with a client. Learn the actual delivery system before prescribing a universal one.

Days 1–30 — Read the environment

ObjectiveActionsEvidence
Understand the FDE systemShadow an active pod; map sponsor, product/platform, ATC, security, data and production ownership.Stakeholder/decision map and engagement flow.
Read one workflow deeplyObserve users, exceptions, current controls and what the process protects.Workflow boundary and exception inventory.
Baseline release mechanicsTrace intake → prototype → evidence → approval → production → run ownership.Current-state load path with friction points.
Build trustContribute directly to workshops, code-with activity, evaluation cases and delivery routines.Accepted increments and explicit feedback.

First deliverables

One Agent Load Path review · one workflow-specific metric baseline · one bounded technical working session · one list of reusable vs client-specific seams.

Days 31–60 — Prove one crossing

ObjectiveActionsEvidence
Ship a thin production sliceChoose a bounded task with real workflow value, explicit human fallback and observable outcomes.Working increment in target environment.
Install the contractsDocument work, context, authority, evidence and run ownership alongside the implementation.Approved release posture and decision record.
Instrument the crossingTrack task success, exceptions, human review, latency, cost and operational recovery.Baseline vs production evidence.
Test portabilityIdentify what can be abstracted across OpenAI, Claude, Vertex/open models and what is platform-specific.Provider seam map and adaptation burden.

Decision at day 60

Recommend one of four postures: release with controls; continue in shadow; human-first assist; hold and redesign. State the evidence and unresolved risk plainly.

Days 61–90 — Leave leverage

ObjectiveActionsEvidence
Operationalize ownershipConfirm on-call/incident path, dashboards, cost unit, change policy, fallback and retirement conditions.Run contract signed by accountable owners.
Package reusable learningHarvest evaluation cases, policy patterns, provider abstractions, workshop prompts and observability conventions.Reusable assets with provenance and adaptation notes.
Strengthen the podCoach engineers and client counterparts through the reasoning, not only the artifact.Client/ATC capability transfer and independent operation.
Shape the next portfolio moveCompare candidate use cases by value, readiness, authority risk, economics and reuse potential.Scale, adapt or stop recommendations.

90-day scorecard

Client outcome advanced · production slice accepted · authority and ownership explicit · metrics operating · reusable learning captured · client/ATC team able to run and improve the mechanism.

Discovery assumptions

This plan is a candidate hypothesis based on public role and company information. Engagement methods, tools and decision rights should be adapted to Accenture's actual delivery standards and the client's environment.
